## 1. Why — 为什么需要 i-Write

### 问题：知识碎片化 + 文档工作繁重

现代知识工作者的信息分散在十几个平台上：

```
┌─────────────────────────────────────────────────────────────┐
│                    你的知识碎片                                │
│                                                              │
│   📧 邮件        💬 Teams 聊天       📄 OneDrive 文档         │
│   📊 Excel 表格   🔀 GitHub PR       📑 arXiv 论文           │
│   📋 会议纪要     📎 本地 PDF         📝 技术文档             │
│                                                              │
│              每天在 10+ 个平台间切换复制粘贴                     │
│              写一份周报要花 2 小时整理信息                       │
└─────────────────────────────────────────────────────────────┘
```

写一份靠谱的文档需要：在多个平台来回切换 → 复制粘贴 → 手动核对 → 担心遗漏。

AI 可以辅助生成，但用户对 AI 生成的内容**缺乏信任**：

```
┌─────────────────────────────────────────────────────────────┐
│              AI 生成文档的信任危机                              │
│                                                              │
│   ❓ 这段话是 AI 编的，还是真的有来源？                         │
│   ❓ 这个数据来自哪份文件？第几页？                             │
│   ❓ 有没有被后续信息推翻？                                    │
│   ❓ 这份文档的质量到底怎么样？能不能直接发出去？                │
│                                                              │
│              用户只能靠肉眼判断 → 不敢直接用                    │
└─────────────────────────────────────────────────────────────┘
```

### 现有方案的不足

```
                    生成能力
                       ▲
                       │
         Gamma         │      ChatGPT / Claude
        (设计生成器)    │      (对话式生成)
                       │      ✅ 能生成  ❌ 不可信
                       │
    ───────────────────┼──────────────────────► 可信度
                       │
         NotebookLM    │      ★ i-Write
        (研究助手)      │      (可信文档生成工作台)
         ✅ 可信        │      ✅ 能生成  ✅ 可信
         ❌ 不生成      │      ✅ 全平台  ✅ 有评估
                       │
```

| 竞品 | 能生成 Office 文档？ | 事实溯源？ | 跨平台知识？ | 离线评估？ | 免费？ |
|------|:---:|:---:|:---:|:---:|:---:|
| ChatGPT / Claude | ✅ | ❌ | 上传文件 | ❌ | 部分 |
| Google NotebookLM | ❌ | 内联引用 | 上传文件 | ❌ | 有限 |
| Microsoft Copilot | ✅ | 生态内 | 仅 MS 生态 | ❌ | ❌ $30/月 |
| Gamma | 仅 PPT | ❌ | ❌ | ❌ | 有限 |
| **i-Write** | **Word+PPT+Excel** | **段落级生成树** | **全平台** | **10+ 指标** | **✅** |

## 2. What — i-Write 是什么

### 一句话定位

> **AI 驱动的文档价值工作台** — 让人类关注文档要传达的信息和故事本身，由 AI 基于你的知识生成文档，你对内容做判断、决策和取舍。

### 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│   传统模式：人类写文档                                        │
│   ────────────────────                                       │
│   收集信息 → 整理素材 → 构思结构 → 遣词造句 → 排版格式        │
│   ⏱️ 耗时 2-4 小时                                           │
│   😩 大量时间花在"写"本身，而非"想"内容                       │
│                                                              │
│                         ↓ 变革                               │
│                                                              │
│   i-Write 模式：人类审文档                                     │
│   ────────────────────                                       │
│   AI 生成初稿 → 人类判断取舍 → 一键定稿                       │
│   ⏱️ 耗时 10-30 分钟                                         │
│   🎯 人类专注内容价值，AI 处理格式、排版、遣词造句             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 人机协作新范式

```
┌─────────────────────────────────────────────────────────────┐
│                     i-Write 人机协作                          │
│                                                              │
│   AI 负责（i = AI）            人类负责                      │
│   ────────────────            ──────────                    │
│   • 连接知识源                 • 决定文档要传达什么            │
│   • 检索相关内容               • 选择哪些观点重要              │
│   • 生成初稿                   • 判断信息是否准确              │
│   • 构建生成树                 • 取舍内容的轻重缓急            │

│   ┌─────────────────────────────────────────────────────┐   │
│   │                                                      │   │
│   │        📄 高质量文档                                  │   │
│   │        人类的判断 + AI 的效率                         │   │
│   │        内容有深度 · 表达有质量 · 来源可追溯            │   │
│   │                                                      │   │
```

## 4. Success Metrics — 成功指标

### 用户价值指标

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 文档生成时间 | 比手动写快 5x+ | 用户从输入需求到获得文档的耗时 |
| 信任度分数 | Groundedness ≥ 0.85 | 每次生成后的在线评估 |
| 用户满意度 | ≥ 4.0 / 5.0 | 文档可直接发出的比例 |
| 知识源利用率 | ≥ 80% | 已连接的知识源被实际引用的比例 |

### 产品质量指标

| 指标 | 目标 | 说明 |
|------|------|------|
| Groundedness | ≥ 0.85 | 生成内容有事实依据的比例 |
| Citation Accuracy | ≥ 0.90 | 引用正确指向来源的比例 |
| Coherence | ≥ 0.85 | 文档逻辑连贯性 |
| Completeness | ≥ 0.80 | 大纲要求的内容覆盖度 |
| Source Coverage | ≥ 0.80 | 知识源被充分利用的程度 |

### 离线评估指标（LLM Model 决策支持）

| 维度 | 指标 | 说明 |
|------|------|------|
| 检索质量 | NDCG@K, Recall@K | 检索结果排序和覆盖 |
| 事实准确性 | Faithfulness, Groundedness | 生成内容是否忠于来源 |
| 引用质量 | Citation Precision, Citation Recall | 引用是否正确和完整 |
| 文档质量 | Coherence, Fluency, Completeness | 文档作为写作的质量 |
| 端到端 | Answer Correctness, Fact Coverage | 与 Golden Set 对比 |

### 评估数据的长期价值

用户积累的评估数据（50+ 次生成）最大的价值是**文档类型适配分析**：

## 9. People Graph — 组织架构与人际关系图谱

### 核心洞察

文档是由人创建的，也是给人看的。人与人之间的关系（上下级、同事、跨部门）**高权重**影响文档的：

- **内容深度**：给老板的汇报 vs 给同事的分享
- **遣词造句**：正式 vs 随意
- **信息取舍**：哪些该详细，哪些该省略
- **行文风格**：直接 vs 委婉

### 场景对比

```
同一个项目进展，两种文档：

给同事的周报：
─────────────
"本周认证模块搞定了，联调通过。遇到个坑，OAuth2 的 redirect_uri 配置问题，花了半天。
下周准备搞支付模块，预计周五能完成。"

给部门 Director 的季度 Review：
─────────────
"本季度用户认证模块已完成开发并通过集成测试，符合 Q2 路线图规划。
过程中解决了 OAuth2 协议兼容性问题，未影响交付时间线。
下一阶段将启动支付系统开发，预计 Q3 中期完成。"
```

**同样的知识，不同的受众，完全不同的文档。**

### 数据来源

| 来源 | 获取方式 | 信号 |
|------|---------|------|
| Microsoft Graph API | 组织架构 API | 汇报关系、部门归属、职级 |
| Teams 聊天记录 | Graph API | 谁和谁经常交流、交流频率 |
| 邮件往来 | Graph API | 谁给谁发邮件、邮件语气 |
| 文档共享 | Graph API | 谁和谁共享文档、协作频率 |
| GitHub | GitHub API | 谁和谁 review PR、协作关系 |
| 用户手动标注 | Chat Box | 用户直接说明"这是给老板看的" |

### 数据结构

```
┌─────────────────────────────────────────────────────────────┐
│  People Graph                                                │
│                                                              │
│  节点: Person                                                │
│  ├─ id, name, email                                          │
│  ├─ department, title, level                                 │
│  └─ communication_style (从历史文档学习)                      │
│                                                              │
│  边: Relationship                                            │
│  ├─ type: reports_to / colleague / cross_team / external     │
│  ├─ strength: 0-1 (基于交流频率)                             │
│  └─ formality: 0-1 (基于历史沟通风格)                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 如何影响文档生成

**1. 受众识别**

```
用户输入: "给 Mike 写一份项目进展汇报"

系统查询 People Graph:
├─ Mike 是谁？→ 部门 Director
├─ Mike 和我的关系？→ 他是我的上级的上级
├─ Mike 的沟通风格？→ 正式、关注数据和 ROI
│
└─ 决策: 生成正式、数据驱动、高层视角的文档
```

**2. 风格自动调整**

| 受众关系 | 内容深度 | 遣词造句 | 信息密度 |
|---------|---------|---------|---------|
| 直属上级 | 中等 | 正式 | 中等 |
| 高层领导 | 战略视角 | 非常正式 | 精简 |
| 同事 | 详细 | 随意 | 详细 |
| 跨部门 | 适度 | 正式 | 中等 |
| 外部客户 | 专业 | 非常正式 | 精炼 |

**3. 知识源优先级调整**

```
给老板写汇报:
├─ 优先使用: 高层数据、里程碑、ROI 指标
└─ 次要使用: 技术细节、实现方案

给同事写分享:
├─ 优先使用: 技术细节、踩坑经验、代码示例
└─ 次要使用: 高层数据、战略背景
```

### 用户交互

**方式一：Chat Box 自动识别**

```
用户: "给 Mike 写一份项目进展汇报"

    ├─ Mike: 部门 Director
```

## 10. Proactive Generation — 基于 Action Item 的主动生成

### 核心场景

```
Teams Meeting 结束
    │
    ├─ Teams 自动生成会议纪要（含 Action Items）
    │   └─ "John to draft feature MVP spec to clarify scope/motivation/metrics"
    │
    ├─ i-Write 检测到 Action Item
    │
    ├─ 自动收集相关知识源
    │   ├─ Teams 聊天记录（之前的讨论）
    │   ├─ 相关文档（PRD、设计文档）
    │   ├─ Excel（数据分析）
    │   ├─ PPT（demo 演示）
    │   └─ 邮件往来
    │
    └─ 主动生成 MVP Specification
        └─ 基于所有相关知识，不是凭空生成
```

### 设计要点

**1. Action Item 解析**

从会议纪要中提取：
- 负责人
- 任务类型（draft / review / analyze / present）
- 文档类型（MVP spec / design doc / report / slides）
- 目标（scope / motivation / metrics）

**2. 智能知识源发现**

根据 Action Item 的关键词，自动搜索相关知识源：
- Teams 频道讨论
- 相关文档
- Excel 数据
- PPT 演示
- 邮件往来

按相关性排序，选择 Top-K。

**3. 质量门控**

```
收集信号 → 内部生成 → Groundedness Check
    │
    ├─ 质量达标 → 建议用户
    └─ 质量不达标 → 继续收集信号，不打扰用户
```

**4. 用户交互**

```
┌─────────────────────────────────────────────────────┐
│  💡 来自 Teams Meeting: "Q3 Feature Planning"        │
│                                                      │
│  Action Item: John to draft feature MVP spec         │
│                                                      │
```

## 11. Workflow — 多步骤文档生成流程

### 核心场景

用户的文档生成往往依赖固定的业务流程，例如"月度业务报告"：

```
Step 1: 从 Excel 读取本月模型结果数据
    ↓
Step 2: 根据数据生成 Pivot Table 做分析
    ↓
Step 3: 引用 Word spec 对数据做说明
    ↓
Step 4: 把文字说明和数据分析的 Pie 图合成 PPT 简报
    ↓
Step 5: 生成 Review 文档
```

很多行业都有类似的固定流程。用户定义一次 workflow，以后反复使用，质量稳定、流程可控。

### 用户定义方式

**方式一：可视化拖拽编辑器**

```
┌─────────────────────────────────────────────────────────────┐
│  Workflow 编辑器 — "月度业务报告"                              │
│                                                              │
│  ┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐      │
│  │Excel│───▶│分析 │───▶│说明 │───▶│PPT  │───▶│Review│      │
│  │数据 │    │Pivot│    │Spec │    │简报 │    │文档  │      │
│  └─────┘    └─────┘    └─────┘    └─────┘    └─────┘      │
│                                                              │
│  每个节点配置:                                                │
│  - 输入来源（文件 / 上一步输出 / 知识库）                     │
│  - 操作（读取 / 分析 / 生成 / 引用）                         │
│  - 输出格式（数据 / 图表 / 文档 / PPT）                      │
│                                                              │
│  [保存 Workflow]  [运行]                                      │
└─────────────────────────────────────────────────────────────┘
```

**方式二：Chat Box 描述**

```
用户: "我每个月需要先跑 Excel 模型，然后根据结果生成分析报告，
      再把分析和 spec 说明合成 PPT 简报"

系统: 识别为 workflow 需求
    → 生成 workflow 草案
    → 用户确认/调整
    → 保存为 workflow 模板
```

### 运行模式

**Reactive（用户触发）**

```
用户点击 [运行] → 选择数据文件 → 自动执行 → 生成所有文档
```

**Proactive（自动触发）**

```
检测到 Excel 数据更新 → 自动运行 workflow → 生成文档 → 通知用户
```

### 步骤间数据传递

```
┌─────────────────────────────────────────────────────────────┐
│  步骤配置                                                    │
│                                                              │
│  步骤名称: 生成 Pivot Table 分析                              │
│                                                              │
│  输入:                                                       │
│  - 上一步输出: model-results (Step 1 的 Excel 数据)          │
│  - 知识库: 相关技术文档                                      │
│                                                              │
│  操作:                                                       │
│  - 分析数据，生成 Pivot Table                                 │
│  - 生成图表 (Pie / Bar / Line)                               │
│                                                              │
│  输出:                                                       │
│  - analysis-result (供下一步使用)                            │
│  - charts (供 PPT 步骤使用)                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 竞品参考

| 产品 | 类似功能 | 借鉴点 |
|------|---------|--------|
| Zapier | 多步骤自动化 | 触发器 + 动作的连接方式 |
| GitHub Actions | Workflow 定义 | YAML 定义 + 可视化编辑 |
| Airflow | DAG 工作流 | 步骤依赖和数据传递 |
| Notion Automations | 自动化流程 | 简单的触发 + 动作配置 |
| LangChain | Chain / Pipeline | 多步 LLM 调用的编排 |

### 核心价值

1. **流程标准化**：固定业务流程一次定义，反复使用
2. **质量可控**：用户对 workflow 的每个步骤都有预期，质量更有信心
3. **效率提升**：不用每次手动执行 5 个步骤，一键完成
4. **知识复用**：workflow 可分享给团队，统一工作流程

### 优先级

P1.5 — 在 P1 基础功能完成后实现，复杂度较高。
